# Support

Thanks for using `drizzle-migration-guard`.

This document explains where different kinds of questions and requests should go so they can be handled in the right place.

## Where to ask

### Bug reports

Open a GitHub issue with the **Bug report** form when:

- the action behaves incorrectly
- PR matching or skip behavior looks wrong
- `drizzle-kit` execution fails in an unexpected way
- markdown summaries, reports, or sticky comments are incorrect

Please include a minimal reproduction whenever possible.

## Feature requests

Open a GitHub issue with the **Feature request** form when:

- you want new behavior or a new input
- you want improved matching, parsing, or reporting behavior
- you want better docs or examples for a real workflow gap

Please describe the problem first, then the proposed behavior.

## Usage questions

For setup and usage questions, check these first:

- [README.md](./README.md) for examples, inputs, outputs, and troubleshooting
- [CONTRIBUTING.md](./CONTRIBUTING.md) for development workflow and validation expectations
- [SECURITY.md](./SECURITY.md) for private security reporting guidance

If the docs do not answer your question, open a GitHub issue and clearly mark it as a usage question. Include the relevant workflow snippet and `drizzle.config.*` shape if configuration is involved.

## Security issues

Do **not** report suspected vulnerabilities in a public issue.

Please follow the private reporting guidance in [SECURITY.md](./SECURITY.md).

## What maintainers can help with

Maintainers are most likely to help with:

- confirmed bugs in repository behavior
- actionable feature proposals with clear scope
- documentation gaps tied to real usage
- regressions affecting released behavior

Maintainers may be slower to respond to:

- broad consulting requests for unrelated CI design
- environment-specific problems without a reproduction
- upstream Drizzle issues that cannot be isolated to this repository

## Response expectations

Support is provided on a best-effort basis.

In general:

- clear, reproducible issues are easier to triage quickly
- smaller, well-scoped requests are easier to act on
- security reports should use the private process in `SECURITY.md`

No SLA is guaranteed, but good-faith reports and questions are welcome.
